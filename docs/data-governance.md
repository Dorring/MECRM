# Phase 6: GDPR + Data Governance

## Goal
Make the CRM compliant, auditable, and safe for regulated environments (EU GDPR, SOC2-aligned practices) without weakening tenant isolation.

## Definitions
### Personal Data (PII)
PII is any data that identifies, relates to, describes, or can reasonably be linked to an individual.

PII classes:
- identity: name, identifiers
- contact: email, phone
- location: address, geolocation
- sensitive: government IDs, payment data
- free_text: unstructured text fields that can contain personal data

Default rule:
- Data is treated as non-PII unless explicitly declared in the PII registry.

### Controllers and actors
- Tenant is the data controller boundary.
- Actor is the entity initiating an action (user, agent, system job).

## Data lifecycle stages
### active
- The row is usable by normal workflows.

### soft-deleted
- The row is logically removed from normal reads.
- The row remains auditable and exportable (DSAR) unless policy states otherwise.
- Soft delete is reversible only by an explicit restore workflow (not part of this phase).

### hard-deleted (GDPR forget)
- PII fields are irreversibly erased (nulled/overwritten) where required.
- Non-PII aggregates may be preserved if legally allowed, but identifiers linking back to the person are removed where possible.
- The action is auditable and includes a reason.

## Supported GDPR rights
### Right to Access (DSAR)
The system supports exporting subject data per tenant:
- customer export: tenant-scoped subject record + related records + audit + AI decisions referencing the subject
- user export: tenant-scoped subject record + related records + audit + AI decisions referencing the subject

Constraints:
- Tenant isolation is absolute (RLS enforced, tenant context required).
- Output is machine-readable JSON.
- Internal-only metadata is excluded or redacted.

### Right to Erasure
The system supports a verifiable forget workflow per tenant:
- forget customer
- forget user

Constraints:
- No silent deletion: every forget produces an audit log entry.
- Soft delete is not considered erasure.
- AI agents must not access forgotten subjects or their PII.

## Retention policy semantics
Retention policies are tenant-specific and specify:
- entity_type
- retention_days
- hard_delete flag

Enforcement model:
- A scheduled job applies policies at rest.
- Expired data is either soft-deleted or hard-erased depending on policy.
- All retention actions are audited.

## AI access restrictions
Agents must respect governance boundaries:
- ignore soft-deleted subjects in decision-making and side effects
- never access or act on GDPR-forgotten subjects

Enforcement:
- A data guard checks deletion state for referenced subjects before agent execution.
- Violations are blocked, metered, and audited.

## Audit and compliance guarantees
Audit logs cover:
- forget requests
- DSAR exports
- retention deletions
- AI access violations

Audit log fields:
- tenant_id
- actor_type, actor_id
- action
- resource_type, resource_id
- old_value, new_value (redacted)
- timestamp
- correlation_id

Guarantees:
- Tenant-scoped storage under RLS.
- Fail-closed behavior when tenant context is missing.
- Deterministic evidence artifacts generated by tests and reports.
